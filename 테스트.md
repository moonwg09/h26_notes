소프트웨어 테스트 (04번 파일)

1. 테스트 개요
-목적 : 결함의 발견을 목적으로 분석하는 활동
-용어


복습용
sw품질(요구사항에 얼마나 잘 따랐는지)
- 프로세스 품질
=> QA
- 제품품질

2. val : 고객의 요구사항 기준


테스트 목적 : 시스템이 정해진 요구사항을 만족하는지 확인하고, 주어진 표준 등을 준수하는지 검증하기 위한 활동
- 결함의 검출과 제품 풀질 개선
- 품질 평가와 의사 결정 지원
- 개발 프로세스 개선 지원

결함 : error, defect, failure
error(오류)로 부터 defect(결함)가 발생하고
defect로 부터 failure(장애)가 발생한다.

error : 사람의 실수
defect : sw내에 장애를 유발할 수 있는 문제(소스 코드가 에러가 난 경우)
failure : defect로 인해 실행될 때 발생되는 이상현상(요구사항과 실제 결과와의 차이가 있다)

오작동은 결함에 의해 발생하지만, 결함이 있다고 반드시 오작동이 발생하는 것은 아님

결함 유형
- 누락 결함 		: 명세서의 내용을 구현하지 않은 경우
- 부정확한 결함 		: 구현은 했지만 부정확하게 구현한 경우 
- 비관련 결함		: 명세서와 관련되지 않은 결함

유지보수 테스트 <=> 테스트 용이성

테스트 원리

1. 테스트는 결함이 존재함을 밝히는 활동
2. 완벽한 테스트는 불가능 ( 적은 양의 노력으로 많은 결함을 발견하는 것)
3. 테스트는 개발초기에 시작
4. 대부분의 결함은 특정 영역에 집중됨(pareto 원칙)
5. 살충제 패러독스: 테스트문서를 업데이트하지 않으면 별로 효과가 없다
6. 테스트는 정황에 의존적:웹페이지 테스트와 자동차엔진 테스트는 다르다
7. 오류부재는 궤변 : val을 강조하는 내용


테스트원칙 중 테스트 수행 그룹의 객관성을 확보해야한다.
=> 나의 소스코드에서 내가 결함을 찾는 것은 어렵다, 개발자와 다른 제 3자가 테스트할 수 있어야한다
